\section*{Phase 12 Wallet Runtime Hardening Plan}

\textbf{Date:} May 16, 2026\\
\textbf{Purpose:} Convert the remaining whole-project gaps into an implementation plan for a mature wallet-aware runtime: first-class wallet/profile/cluster state, organic live proof, production delivery proof, evidence benchmarking, larger shadow-ML evidence, and SSD-backed replay.

\section*{Executive Target}
The target is \textbf{not} to turn the project into an autonomous trading system. The target is to make the existing local-first research/runtime system mature enough that it can honestly claim:
\begin{itemize}[nosep]
\item wallet entities are first-class tables, not only nullable fields on trades,
\item live runtime evidence comes from an organic post-prune run, not only a bounded synthetic proof,
\item alert delivery is proven through at least one real outbound provider or a clearly documented provider-disabled fallback,
\item evidence retrieval quality is benchmarked instead of assumed,
\item shadow ML evidence is expanded beyond the current small held-out packet,
\item archived replay can run from the attached SSD without copying the full archive back to internal disk,
\item CLOB V2 compatibility is explicitly checked before new organic runtime proof.
\end{itemize}

\section*{Implementation Principles}
\begin{enumerate}[nosep]
\item Keep rule-based Phase 4 alerts authoritative; ML remains shadow-only until a later promotion memo.
\item Build wallet identity as public-wallet intelligence only. Never imply real-world identity.
\item Prefer idempotent materializers over one-off notebooks or manual SQL.
\item Every new claim needs a machine-readable report under \texttt{reports/phase12/}.
\item Keep raw archives and detector-input archives off git; rely on SSD mount or explicit restore for heavy replay.
\item Do not restart long-running collection until schema changes, storage checks, and delivery settings are known-good.
\end{enumerate}

\section*{Workstream A: First-Class Wallet/Profile/Cluster Plane}
\textbf{Goal:} Promote wallet information from trade columns into durable entity tables with materialized activity/profile/cluster state.

\subsection*{A1 Schema Additions}
Add idempotent schema blocks to \texttt{database/schema.sql} and \texttt{database/postgres\_schema.sql}:
\begin{itemize}[nosep]
\item \texttt{wallets}: one row per public wallet/proxy wallet.
\item \texttt{wallet\_first\_seen}: first observed trade/event provenance.
\item \texttt{wallet\_activity}: rolling and lifetime activity aggregates.
\item \texttt{wallet\_profiles}: optional public profile/API snapshot state.
\item \texttt{wallet\_positions}: optional current or sampled market positions.
\item \texttt{wallet\_cluster\_membership}: reversible statistical cluster labels.
\item \texttt{wallet\_entity\_runs}: materialization run ledger.
\end{itemize}

\textbf{Minimum columns:}
\begin{itemize}[nosep]
\item \texttt{wallet\_id}, \texttt{proxy\_wallet}, \texttt{first\_seen\_at}, \texttt{last\_seen\_at}, \texttt{source\_count}, \texttt{created\_at}, \texttt{updated\_at}.
\item Aggregate fields: \texttt{trade\_count}, \texttt{market\_count}, \texttt{condition\_count}, \texttt{notional\_total}, \texttt{notional\_24h}, \texttt{notional\_7d}, \texttt{avg\_trade\_size}, \texttt{last\_trade\_at}.
\item Profile fields: \texttt{profile\_source}, \texttt{profile\_status}, \texttt{profile\_json}, \texttt{profile\_captured\_at}.
\item Cluster fields: \texttt{cluster\_id}, \texttt{cluster\_version}, \texttt{method}, \texttt{confidence}, \texttt{features\_json}, \texttt{assigned\_at}.
\end{itemize}

\subsection*{A2 Materialization CLI}
Create \texttt{run\_wallet\_entity\_materializer.py}. It should:
\begin{enumerate}[nosep]
\item read \texttt{canonical\_trades} where \texttt{proxy\_wallet} is not null,
\item upsert \texttt{wallets} and \texttt{wallet\_first\_seen},
\item compute aggregate rows into \texttt{wallet\_activity},
\item write a \texttt{wallet\_entity\_runs} row,
\item emit \texttt{reports/phase12/wallet\_entity\_materialization.json}.
\end{enumerate}

\textbf{Acceptance criteria:}
\begin{itemize}[nosep]
\item command exits 0 on SQLite,
\item command can run twice without duplicating rows,
\item report includes \texttt{wallet\_count}, \texttt{wallets\_with\_first\_seen}, \texttt{activity\_rows}, null-rate metrics, and top-wallet aggregate sanity checks,
\item \texttt{run\_runtime\_status.py} or a new Phase 12 status command reports wallet-plane health.
\end{itemize}

\subsection*{A3 Profile and Position Enrichment}
Create a second CLI, \texttt{run\_wallet\_profile\_refresh.py}, after A2 is stable. It should be budgeted and optional:
\begin{itemize}[nosep]
\item select top-N wallets by recent notional or alert involvement,
\item call configured public profile/position endpoints only when enabled,
\item cache every response with timestamp, provider, status, and raw metadata,
\item fail soft when provider credentials or rate budgets are unavailable.
\end{itemize}

\textbf{Acceptance criteria:} at least one cached profile/position refresh packet exists, or the report explicitly says provider disabled with no false success.

\subsection*{A4 Cluster Baseline}
Start with transparent heuristic clusters, not graph ML:
\begin{itemize}[nosep]
\item cluster by repeated co-occurrence in the same markets/conditions,
\item use common active-window overlap and notional concentration,
\item write \texttt{wallet\_cluster\_membership} with \texttt{method=heuristic\_v1},
\item include enough features in \texttt{features\_json} to audit why the assignment happened.
\end{itemize}

\textbf{Acceptance criteria:} cluster assignments exist for wallets with sufficient overlap, and docs clearly call them statistical groups, not people.

\section*{Final Claim Boundary}
